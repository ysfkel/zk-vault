## ZK Vault

ZK Vault provides private withdrawals using zero-knowledge proofs. Deposits create commitments included in an on-chain Merkle tree; withdrawals present ZK proofs that a commitment exists and a nullifier is unused to prevent double-spends.

Highlights
- Private withdrawals using zero-knowledge proofs and nullifiers.
- Merkle tree depth: 20 (proofs include 20 sibling nodes).
- Root history: a ring buffer (default 30 roots) — older roots are overwritten as new ones are added.

Repository layout
- `circuit/` – Noir circuit sources and compiled artifacts (`circuit/target/circuit.json`).
- `contracts/` – Solidity contracts and tests (vault, verifier, incremental Merkle tree).
- `contracts/js-scripts/` – helper scripts used by tests (`generate_proof.ts`, `merkle_tree.js`).

Requirements
- Foundry (`forge`) for compiling & running tests.
- Node.js + npm for proof helper scripts (tests call these via `vm.ffi` using `npx tsx`).
- Noir / Barretenberg toolchain to build the circuit and generate the Solidity verifier.

Quickstart
1. Install Foundry and Node.js.
2. Install contract dependencies (recommended Forge flow):

```bash
cd contracts
forge install
cd js-scripts
npm install
```

3. Build the Noir circuit and generate the Solidity verifier

```bash
cd ../../circuit
nargo build --release
bb write_solidity_verifier -k ./target/vk -o ./target/Verifier.sol -t evm
```

4. Compile contracts and run tests

```bash
cd ../contracts
forge build
forge test -vv
```

Running proof helper scripts manually

To debug proof generation directly (outside `vm.ffi`):

```bash
npx tsx contracts/js-scripts/src/generate_proof.ts <nullifier> <secret> <recipient> <leaf1> <leaf2> ...
```

Troubleshooting
- "leaf not found": ensure leaves are provided as 0x-prefixed hex strings — `merkle_tree.js` stores leaves as hex strings and `generate_proof.ts` expects normalized 0x-hex inputs.
- `ProofLengthWrongWithLogN(...)` revert: indicates a prover/verifier mismatch. Regenerate the Solidity verifier from the same circuit artifacts used by the prover (see step 3) and recompile contracts. Add a debug print in `generate_proof.ts` to confirm the generated proof byte length.
- Debug prints from FFI-run scripts may be suppressed by the VM; run the script directly as shown above or write logs to stderr to ensure visibility.

Developer notes
- The circuit uses a Merkle depth of 20. Confirm the on-chain `Verifier.sol` is generated from the same circuit and parameters (LOG_N / N) as the prover.
- The `contracts/js-scripts` helpers normalize hash formats; mismatches between Uint8Array and 0x-hex strings are a common cause of "leaf not found" errors.

Secret, Nullifier, and Commitment
- Secret: a private random value generated by the depositor. It is never revealed on-chain. The secret is used so the commitment cannot be guessed or correlated by observers.
- Nullifier: a private random value (one per deposit) used to derive a public `nullifier_hash`. The nullifier itself is kept private inside the ZK proof; only `nullifier_hash` (a one-field Poseidon hash of the nullifier) is exposed to the contract when withdrawing.
- Commitment: the public value inserted as a leaf in the Merkle tree. In this project the commitment is computed as a two-input Poseidon hash of `(nullifier, secret)` (order matters):

	commitment = Poseidon2(nullifier, secret)

	and the nullifier hash (public input) is:

	nullifier_hash = Poseidon2(nullifier)

Why only the commitment is stored as leaves
- Storing only the commitment in the Merkle tree preserves privacy: the contract and tree do not learn the `secret` or `nullifier` values, only a short public commitment that proves the deposit happened.
- When withdrawing, the prover supplies a ZK proof that it knows `nullifier` and `secret` such that `commitment = Poseidon2(nullifier, secret)` and that `commitment` is included in the Merkle tree (Merkle proof). The proof also asserts that the provided `nullifier` hashes to the public `nullifier_hash` (so the contract can check and mark that nullifier as spent).
- The contract records `nullifier_hash` on successful withdrawal to prevent reuse (double-spend) without learning `nullifier` or `secret` themselves.

Implementation notes
- The circuit (`circuit/src/main.nr`) computes `commitment = poseidon2::Poseidon2::hash([nullifier, secret], 2)` and `computed_nullifier_hash = poseidon2::Poseidon2::hash([nullifier], 1)` and checks `nullifier_hash == computed_nullifier_hash` before verifying Merkle membership.
- The JS helper `contracts/js-scripts/src/generate_commitment.ts` uses Barretenberg's `poseidon2Hash` with inputs `[nullifier, secret]` and returns `commitment.hash` (bytes32) plus the raw buffers for `nullifier` and `secret` when needed for tests.
- Important: input order matters to Poseidon; use the same ordering in prover and any off-chain helpers.
